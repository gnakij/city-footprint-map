# 城市足迹地图 · 设计与界面指南

> 最后更新：2026-07-20

## 1. 设计目标

界面应像一份安静、温暖、可以长期使用的旅行记录册。地图和个人回忆是主角，管理能力退居次要位置。

设计判断遵循以下顺序：

1. 内容是否清楚；
2. 操作是否容易找到；
3. 桌面端和移动端是否都能顺畅使用；
4. 五套主题下是否保持一致；
5. 是否复用了现有组件和语义变量。

不要为了显得“功能丰富”增加装饰、说明卡片或重复入口。

## 2. 视觉层级

页面由四个主要层级组成：

- 顶部栏：品牌、当前用户和账号菜单；
- 地图：最主要的浏览与记录区域；
- 足迹统计：重要但可收起的辅助面板；
- 弹窗与抽屉：个人资料、城市记录和系统管理。

地图应始终保留足够空间。桌面端统计面板位于右侧，地图控件位于左侧；移动端统计面板进入地图下方，控件仍保持左侧，避免同一功能在不同设备上改变方向。

## 3. 设计变量

设计变量的唯一实现来源是：

- `src/styles/tokens.css`；
- `src/styles/base.css`；
- `src/styles/map.css`；
- `src/styles/admin.css`；
- `src/styles/responsive.css`。

新增样式应优先使用语义变量，不直接写具体颜色。常用变量包括：

| 变量 | 用途 |
|---|---|
| `--color-primary` | 主按钮、当前状态和品牌强调 |
| `--color-on-primary` | 主色背景上的文字 |
| `--color-background` | 页面基础背景 |
| `--color-surface` | 面板和弹窗背景 |
| `--color-surface-container-low` | 输入框、次级容器和悬停状态 |
| `--color-on-surface` | 主要文字 |
| `--color-on-surface-variant` | 次要文字 |
| `--color-outline-variant` | 边框和分隔 |
| `--color-error` | 删除、清空等危险操作 |
| `--color-focus-ring` | 键盘焦点和输入焦点 |
| `--radius-control` | 输入框和紧凑控件 |
| `--radius-panel` | 面板、卡片和弹窗 |
| `--size-btn-default` | 常规按钮高度 |
| `--size-input` | 输入框高度 |

颜色调整必须同时检查五套主题，不能用固定蓝色、粉色或红色覆盖某个局部问题。

## 4. 字体与文字

使用设备系统中文字体，不依赖外部字体服务。

- 正文使用常规字号和字重；
- 页面标题只在弹窗或独立页面使用；
- 面板标题应比页面标题更小；
- 表头可以加粗，但颜色应弱于正文；
- 次要说明使用 `--color-on-surface-variant`；
- 按钮使用简短动作文字，不在按钮中放操作说明；
- 升级记录使用面向使用者的语言，不写代码类名、排查过程或内部讨论。

移动端不能通过视口宽度动态缩放字体，长文字应自然换行。

## 5. 组件规范

### 按钮

- 主按钮：当前区域最重要的确认或创建动作；
- 描边按钮：导入、导出、取消等次要动作；
- 文字按钮：表格和列表中的低强调操作；
- 危险按钮：删除、清空等不可逆操作；
- 图标按钮：关闭、缩放、重置等熟悉动作，并提供可访问名称。

同一区域通常只保留一个主按钮。

### 输入框

输入框应有明确的默认、只读、编辑、聚焦、错误和禁用状态。用户名等不可修改字段使用只读样式，不使用禁止鼠标指针制造额外压力。

日期、数字和选择类输入应使用对应控件，不用自由文本模拟。

### 表格

- 表头与明细必须有清晰的背景和字重层级；
- 数据较多时由表格区域内部滚动；
- 表头保持可见；
- 行内操作使用低强调文字按钮；
- 桌面端使用表格，移动端在内容过宽时改用卡片；
- 列表超过单页容量时显示分页。

### 弹窗与抽屉

- 标题和关闭按钮固定在顶部；
- 内容在弹窗内部滚动；
- 相邻页签使用稳定的内容高度，避免切换时跳动；
- 数据较少时允许留白，不用无意义装饰填满空间；
- 移动端弹窗保留安全边距，不遮挡系统手势区域。

### 地图控件

重置、放大和缩小组成一个轻量工具组。桌面端保持紧凑，移动端触控区域可以更大，但图标、顺序和方向保持一致。

地图颜色必须通过 `src/utils/colors.ts` 读取主题变量。修改地图数据或缩放逻辑后，应同时测试全国视图、省级视图、鼠标滚轮、拖拽和双指缩放。

### 足迹统计

统计面板是核心内容，不属于普通工具卡片。它应突出关键数字、成就与排行，同时避免遮挡地图。收起与展开过程不能改变面板左右位置。

## 6. 响应式规则

项目主要断点：

- 767px 及以下：移动端；
- 768px 到 1024px：平板与窄桌面；
- 1024px 以上：桌面端。

响应式修改必须至少检查：

- 1366 × 900 桌面视口；
- 1024px 左右窄桌面；
- 412px 左右手机视口；
- 页面内容较多和较少两种状态。

不要只缩小桌面布局。移动端需要重新安排表格、弹窗和统计区域，但同一功能的图标、名称和交互含义应保持一致。

## 7. 动效与反馈

动效用于说明状态变化，不用于装饰。

- 点击后立即提供加载反馈；
- 收起、展开和弹窗出现使用短时过渡；
- 避免改变宽度或左右定位造成跳动；
- 支持 `prefers-reduced-motion`；
- 操作成功或失败使用短暂提示；
- 提示不能长期遮挡标题、输入框或主要操作。

## 8. 可访问性

- 所有功能都应可通过键盘完成；
- 图标按钮必须有 `aria-label` 或等价名称；
- 弹窗打开后焦点进入弹窗，关闭后回到触发按钮；
- 菜单支持方向键和 Escape；
- 文字与背景保持足够对比；
- 不只依赖颜色区分错误、选中或危险状态；
- 触控按钮保留足够点击面积。

## 9. 扩展原则

新增功能前先确认：

1. 是否属于城市记忆的核心流程；
2. 是否已有相近入口或组件；
3. 是否需要普通用户看到；
4. 是否适配五套主题；
5. 是否同时考虑桌面端和移动端；
6. 是否需要补充加载、空状态、错误状态和测试。

共享组件只承担通用行为，不依赖本项目专属样式。业务组件应继续使用项目自己的 class 和语义变量。

## 10. 发布前界面检查

- 登录页没有主题闪烁或额外中转画面；
- 地图、控件和统计面板不互相遮挡；
- 五套主题下表格、输入框和按钮层级清楚；
- 移动端标题、按钮和卡片文字不溢出；
- 弹窗标题固定，内容区独立滚动；
- 升级记录和系统文档内容简洁、日期正确；
- 桌面端和移动端自动检查通过，并进行一次人工截图复核。
