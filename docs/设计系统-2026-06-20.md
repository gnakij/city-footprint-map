# 城市足迹地图 · 设计系统

> 关联文件：`src/index.css`（token 定义）、`CLAUDE.md`（UI 规范）

---

## 一、设计理念

### 1.1 品牌定位

| 维度 | 定义 |
|------|------|
| **目标用户** | 旅行爱好者，记录去过的城市 |
| **情感调性** | 温馨旅行日记感 |
| **视觉隐喻** | 日记本、邮票、地图 |
| **设计原则** | ①温馨不花哨 ②地图边框硬编码不动 ③字重不改 ④页签方形非胶囊 |

### 1.2 设计原则（决策准则）

1. **温馨优先**：圆角、柔和阴影、暖色调，避免冷硬直角
2. **一致性**：所有组件引用 design token，不硬编码数值
3. **可维护**：变量化 > 硬编码，改一处全局生效
4. **克制**：不随意添加功能，新增内容融入现有设计

## 四、地图规范（特殊）

### 4.1 硬编码规则

以下属性**硬编码，不跟随主题**：

| 属性 | 值 | 原因 |
|------|------|------|
| 省界轮廓 `borderColor` | `#666666` | 移动端反馈"太粗发黑"，从 `#B98A98` 调整 |
| 省界轮廓 `borderWidth` | `0.5` | 细灰色描边 |
| 地图标签 `fontWeight` | 不改 | 用户偏好细体 |

**2026-06-20 调整**：用户反馈"地图很多地方没按统一规范走，换主题后悬停色
还是黄色、点亮城市字体还是红色，看起来很乱"。复核后确认地图悬停预览色
（曾硬编码 `#FFD166`）和点亮城市标签色（`LABEL_COLOR_LIT`/`LABEL_COLOR_UNLIT`，
独立于上面这张"硬编码不跟主题"的表）此前确实都是固定值，不随主题切换——
但这两处**不属于本节"刻意保留的硬编码"**（本节列出的三项都有明确的、记录过
的设计理由：移动端反馈、用户偏好），是当时实现时没有考虑到多主题场景遗留
下来的。已将悬停色改为复用 `--color-warning`（语义贴切，此前未被使用），
三个主题各自定义专属值（Rose `#FFB74D`、Linear `#FFC857`、Stripe `#F5A623`，
经验证不会与各主题自身的染色梯度色撞色）。

**2026-06-20 已修复**：点亮城市/省份标签色同样改为跟随主题。用户明确要求
"已点亮统一用品牌主色，未点亮用现有灰色调，不新造颜色、不要衍生变体"——
最终方案比预想的更简单：已点亮直接复用 `--color-primary`，未点亮直接复用
`--color-on-surface-variant`，两个变量三主题本来就已经各自定义好协调的值。
验证主色直接用作标签色对地图背景的对比度：Rose 4.79:1、Linear 4.24:1、
Stripe 6.19:1，均可读（Linear略低于4.5是其主色本身偏暗的固有特性）。
未点亮色此前已验证对比度均≥4.5:1（见上方"地图染色梯度"小节附近说明）。

**2026-06-20 已修复**：地图标签弹窗(tooltip)的边框色，原本是 ECharts 默认
行为——取被点击数据项本身的填充色(`tooltipDataParams.color`)作为边框色，
导致边框颜色随城市染色梯度变化(停留天数落在`--color-dur-l2`/`l3`等中间档
时呈现蓝色，与主题色调不符)。用户要求固定使用主题色，已通过显式设置
`tooltip.borderColor`（复用`--color-primary`）覆盖了这个默认行为——已用
ECharts 源码确认该配置项会被直接拼接进最终tooltip元素的`border-color`
内联样式（`TooltipHTMLContent.js`的`show()`方法），不是只影响内部计算
逻辑、不生效于视觉效果。

### 4.2 性能优化

- `devicePixelRatio` 钳到 2，避免高清屏过度渲染
- 双指 pinch 缩放+平移走 CSS transform（GPU 合成）
- 触摸期间屏蔽 hover 预览（`isTouchingRef` 标志位）

## 五、可访问性规范

### 5.1 Focus Ring

所有可交互元素必须有 `:focus-visible` 样式：

```css
:focus-visible {
  outline: 2px solid var(--color-focus-ring);
  outline-offset: 2px;
}
```

**待确认事项**：`--color-focus-ring` 当前定义为 `rgba(0,102,255,.12)`，
透明度只有 0.12，理论上颜色本身会非常浅。规则本身确实生效（CSS 选择器和
属性都对），但"是否真的能被用户清楚看到"是视觉效果问题，无法用代码静态
判断，需要实际在页面上按 Tab 键走一遍焦点顺序，肉眼确认这个焦点环是否
足够明显——如果实测发现不够清晰，需要调高这个变量的透明度或换用更深的色值。

### 5.2 颜色对比度

遵循 WCAG AA 标准：
- 正文文字：对比度 ≥ 4.5:1
- 大标题（≥18px 或 ≥14px bold）/图标：对比度 ≥ 3:1

**2026-06-20 复核说明**：本节此前的数值未经实际计算，凡三主题填同一个数字
的行基本可以确认是估算/誊抄而非真算（同一对语义变量在三套主题里取的是不同
色值，结果不可能完全相同）。现已用标准 WCAG 相对亮度公式重新计算，并标注
每个变量当前是否真的被组件使用——不达标但未被使用的变量目前没有实际可见性
风险，已用单独说明列出，区别于"不达标且正在被使用、需要处理"的情况。

**需要关注的具体问题**（按当前实际影响排序，已修复项见上方说明）：

1. **Rose/Stripe 主题"错误色 on 表面"3.91:1 / 4.85:1**——Rose 不达标，Stripe
   达标。`--color-error` 被用在 `.danger-text` 和 `.error-state` 的正文文字
   上，Rose 主题下这部分文字理论上不满足 AA 标准，但实际是否构成问题取决于
   这两个 class 目前在产品里出现的频率和场景（弱提示 vs 关键操作警示），
   建议人工过一遍这两个 class 实际渲染的位置再判断是否需要改色值。
2. **次要文字 on 表面**（`.label-sm`、空状态文字等大量使用的辅助文字色）
   Rose 4.34:1、Stripe 4.47:1，**两者都略低于** 4.5:1 的 AA 正文门槛，差距
   很小（0.03~0.16），实际视觉上大概率察觉不出问题，但严格按标准判定是
   不达标的，记录在此供后续参考。
3. `--color-success`/`--color-warning` 三个主题对比度普遍偏低（Rose/Stripe
   只有2.0~2.6:1，连图标用途的3:1门槛都不够），**但这两个变量目前在全部
   组件代码里都没有被实际引用**，是定义了但未使用的"孤儿变量"——暂无可见性
   风险，但如果以后真的要启用这两个语义色，需要先重新选色，不能直接套用
   现在定义的色值。

**之前文档里的错误，记录下来避免下次重蹈覆辙**：原表格三套主题在多行都填了
完全相同的数字（如"主要文字 on 表面"三主题统统标 17.40:1），这在数学上是
不可能的——三套主题的文字色和背景色都不同，对比度计算结果理应各不相同。
这是个比单纯数值算错更需要警惕的信号：**填三遍相同数字基本意味着没有真的
针对每个主题分别计算**，以后任何"理应不同却写了相同值"的表格都要重新核实。

## 六、响应式断点

项目使用三个断点（CSS @media 不支持变量引用，断点值直接写在媒体查询中）：

| 名称 | 数值 | 用途 | 媒体查询 |
|------|------|------|----------|
| 移动端 | ≤767px | 手机布局 | `@media (max-width: 767px)` |
| 桌面端 | ≥768px | 桌面布局 | `@media (min-width: 768px)` |
| 平板 | 768-1024px | 平板适配 | `@media (min-width: 768px) and (max-width: 1024px)` |

**响应式策略**：
- Mobile First：先写移动端，用 `min-width` 媒体查询增强
- 流式布局：`flex` / `grid` + `clamp()` + 相对单位
- 内容优先级：移动端隐藏次要内容

## 七、全局状态规范
